TASK
Create a concise, ATS-oriented CV tailored to the job. Use the risk-control
strategy to decide what to foreground, but never invent facts. Treat every
do_not_claim item as a hard exclusion.

PROJECT SELECTION
Choose Projects by matching the job's must-have and good-to-have requirements,
not by always promoting the same strongest-looking project. No project is
mandatory.

Use this priority:
1. Projects that directly prove the job's core role type and technologies.
2. Projects that prove adjacent transferable work.
3. Omit projects that distract from the target role, even if they are strong.

Examples:
- Mobile App Developer / iOS Developer / Android Developer / Flutter role:
  prioritize AirEye or other mobile app projects, Swift/iOS, Kotlin/Android,
  Flutter, release automation, app security, Firebase, Fastlane, and store
  deployment work. Do NOT include AirEye-Cluster unless the JD explicitly asks
  for Kubernetes, cloud infrastructure, DevOps, platform engineering, backend
  infrastructure, observability, or LLM infrastructure.
- Backend / Full-stack role:
  prioritize API, database, authentication, PostgreSQL/MariaDB, Node.js,
  TypeScript, Spring Boot, Django/Python-adjacent backend evidence, and fullstack
  projects. Include AirEye-Cluster only when infrastructure, Kubernetes,
  observability, cloud, or LLM/backend platform work is relevant.
- DevOps / SRE / Platform / Cloud role:
  prioritize AirEye-Cluster, Terraform, Kubernetes, GitOps, Vault, Keycloak,
  observability, CI/CD, and cloud infrastructure.
- AI / LLM infrastructure role:
  prioritize AirEye-Cluster only for LiteLLM, Langfuse, multi-provider routing,
  tracing, observability, and platform integration evidence. Do not call it a
  production agentic system.

If a role only needs 1-2 highly relevant projects, return only those projects.
Avoid filling the Projects section with unrelated work.

Return EXACTLY one JSON object and nothing else. No prose, no markdown fences,
no preamble.

RULES
- Use only facts supported by MASTER CV.
- Do not add claims blocked by risk_control.do_not_claim.
- Use risk_control.safe_to_enhance and risk_control.cv_strategy only when the
  MASTER CV supports the wording.
- Keep bullets short, specific, and action/result oriented.
- Reorder and rephrase real bullets for relevance, but do not fabricate metrics,
  employers, degrees, dates, certifications, frameworks, or production claims.
- Each included work experience entry must contain at least 5 truthful bullets.
  If the job needs a narrower angle, keep the most relevant 5+ bullets instead
  of pruning the role down to only exact keyword matches.
- Do not remove supported engineering, automation, production support,
  cross-team coordination, mobile delivery, CI/CD, backend, or infrastructure
  bullets solely because they are not exact JD keywords.
- competencies should contain 6-10 phrases.
- experience should preserve real employers and dates from the MASTER CV.
- projects should include only real projects from the MASTER CV, selected by
  direct relevance to the JD requirements. Include 1-4 projects, but fewer is
  better when only a small number are relevant. If risk control says a new
  project should be built, do not pretend it already exists.
- language must be "en".
- paper_format should be "letter" only for US or Canada jobs; otherwise "a4".
- Do not add keys outside the schema.
